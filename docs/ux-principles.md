# UX 원칙 — 중복 기능 / 일관성 / 견고함 기준

> 챗봇 UI 가 커지면서 같은 기능을 여러 페이지에서 보여줘야 할 때가 생깁니다.
> 이 문서는 **어떤 중복은 허용**하고 **어떤 중복은 제거**하는지의 기준을 정립합니다.
> 새 페이지/컴포넌트 만들 때 PR 리뷰의 기준선이 됩니다.

## 1. 중복 허용 기준 (Convenient Redundancy)

다음 조건을 **모두** 충족하면 중복은 사용자에게 도움이 됩니다.

1. **빠른 경로(shortcut) 역할** — 본 위치에 가지 않아도 같은 결과를 얻음
2. **동일한 라벨 + 동일한 결과** — 어디서 누르든 같은 동작
3. **호출 횟수가 잦음** — 매일/세션마다 여러 번
4. **현재 컨텍스트와 자연스러움** — 그 페이지의 맥락 안에서 손에 닿는 위치

**허용 사례** (이 챗봇에 적용):

| 기능 | 본 위치 | 중복 위치 | 이유 |
|---|---|---|---|
| 로그아웃 | `/admin` 우상단 | 모든 page top-bar 우측 | 어디서든 즉시 종료, 1-click |
| 테마 변경 | top-bar 우측 테마 갤러리 (`BrandThemeGallery`) | 첫 화면·`/manual` 의 테마 갤러리 | 즉시 시각 피드백 — 페이지 이동 비용 0 |
| 새 챗 만들기 | `/chatbots` | `/chat` 좌측 사이드바 | 채팅 중 다른 봇으로 전환 가능 |
| 사용자 메뉴 (프로필/로그아웃) | 헤더 아바타 | (이 chatbot 은 단일 위치) | 단일 위치 유지 |
| 검색 | (없음) | 향후 도입 시: top-bar (모든 페이지) | 어디서든 시작 |

## 2. 중복 금지 기준 (Confusing Redundancy)

다음 중 하나라도 해당하면 중복은 사용자에게 해롭습니다.

1. **결과가 미묘하게 다름** — "저장" 버튼 2개인데 한쪽은 임시저장, 한쪽은 게시
2. **같은 페이지 안 같은 액션 2회** — 폼 위·아래 둘 다 "제출"
3. **라벨은 같은데 적용 범위가 다름** — "삭제" 가 한 곳은 선택만, 한 곳은 영구
4. **호출 횟수가 드묾** — 1년에 1~2회 쓰는 액션을 여러 위치에 두면 인지 부담만 늘림

**금지 사례** (적용 시 fail):

| 잘못된 패턴 | 왜 나쁜가 |
|---|---|
| 같은 페이지에 "취소" 버튼 2개 | 어느 버튼이 변경 되돌리고 어느 게 페이지 떠나는지 불명확 |
| 채팅 페이지에서 "스케줄 등록" 직접 버튼 + 별도 페이지에 "스케줄 등록" | 등록 흐름이 둘로 나뉘면 한쪽이 항상 stale (현재 우리 챗봇: 채팅에서 옵션 카드만, 등록은 백엔드 → 페이지는 조회만 → OK) |
| 사이드바·헤더 양쪽에 같은 "도구" 링크 | 시각적 노이즈 |
| 모달 안에서 "닫기" 와 "취소" 두 라벨로 같은 동작 | 한 가지 라벨로 통일 |

## 3. 라벨링 원칙

1. **동사 + 목적어** (예: "스킬 만들기", "테마 적용") — 명사만 두지 않음 ("스킬", "테마" 단독 금지)
2. **존댓말 일관성** — "삭제", "저장" 같은 명령어는 짧게. 사용자 안내문은 존댓말 "~해 주세요"
3. **금지 문구**:
 - "확인" (Yes/No 어느 쪽인지 불명확) → "삭제할게요" / "취소"
 - 이모지로 의미 전달 (의미를 단어로 보강 — 시각장애 + 색맹 사용자 대응)

## 4. 시각/공간 원칙

1. **여백** : 카드 사이 최소 16px, 카드 내부 패딩 24px 이상 (data-dense 페이지 제외)
2. **글자 크기** : 본문 14~16px, 제목 H1 28~32px, H2 20~22px. 한글 가독성 우선
3. **컬러 토큰** : 직접 hex 금지. `var(--colors-*)` 사용 → BrandThemeProvider 가 브랜드별로 자동 매핑
4. **반응형** : `sm: 640px` 이상에서 사이드바, 미만에선 햄버거 또는 bottom tab
5. **로딩 상태** : 절대 빈 화면 + 텍스트만 두지 않음. spinner 또는 skeleton 사용
6. **에러 상태** : 한국어 문장 + 다음 조치(재시도/돌아가기) 버튼

## 5. 견고성 원칙

1. **모든 API 호출은 `apiFetch` 통과** — 직접 `fetch` 금지. 자동 인증 + 에러 처리 + 로깅
2. **에러는 `ApiUserError`** — `requestId`, `path` 보존 → backend 로그 추적 가능
3. **컴포넌트가 에러나도 페이지 전체 안 죽기** — `error.tsx` 가 잡음
4. **레이아웃 시프트 없음** — fetch 결과로 컴포넌트 크기가 갑자기 바뀌지 않게 (skeleton 으로 자리 잡음)
5. **이모지 사용 금지** — UI 라벨/버튼에 이모지 안 씀 (가독성/접근성/검색성). 데이터 출처 표시는 텍스트 또는 색 도트로

## 6. 의사결정 흐름 (체크리스트)

새 기능 추가 전 자문:

- [ ] **언제 호출되는가** — 매일/세션? 1번? 1년에 1번? → 빈도가 낮으면 단일 위치
- [ ] **현재 페이지 컨텍스트와 자연스러운가** — 채팅 중 챗봇 전환은 자연. 채팅 중 사용자 권한 변경은 어색
- [ ] **다른 곳에도 있나** — 있다면 라벨/결과 일치 검증
- [ ] **에러 시 어떻게 보이나** — 빈 화면? 에러 메시지? 재시도 버튼?
- [ ] **느린 응답 시** — spinner? skeleton? 점진적 표시?
- [ ] **이모지 안 썼나** — 텍스트 라벨로 충분히 명확한가
- [ ] **컬러/여백 토큰만 썼나** — `var(--colors-*)`, `var(--radius-*)` 등

## 7. 적용 매트릭스 (현재 챗봇)

> 운영(`NEXT_PUBLIC_FEATURE_SET=rag`) 빌드는 채팅·챗봇 만들기·자료 올리기·문서 검색 4기능만 노출하고, 나머지는 **개발 전용(운영 미노출, ADR-0009)** 이다. `/tools`·`/skills`·`/schedules` 는 `isRagOnly` 가드로 메뉴·페이지가 숨겨지고 운영 백엔드는 해당 라우터를 등록하지 않는다(404).

| 페이지 | 본 책임 | 중복 / 빠른 경로 | 검토 |
|---|---|---|---|
| `/chat` | 대화 | 좌측: 챗봇 전환·새 대화 (빠른 경로 OK) | 통과 |
| `/chatbots` | 챗봇 CRUD | (단일 위치) | 통과 |
| `/documents` | 문서 업로드/관리 | (단일 위치) | 통과 |
| `/schedules` (개발 전용) | 예약 조회 | 등록은 채팅에서만 — 의도적 단일 진입 | 통과 |
| `/tools` (개발 전용) | 도구 마켓 + 자격증명 | (단일 위치) | 통과 |
| `/skills` (개발 전용·구현됨) | 스킬 마켓 | 본 위치 등록, 어드민이 org 승격 | 통과 |
| `/team` | 팀원 + 감사 | (단일 위치) | 통과 |
| `/admin` | 권한·메트릭 | top-bar 메뉴에서 진입 | 통과 |
| top-bar | 테마 / 로그아웃 / 알림 | 모든 page 공통 | 통과 (빠른 경로) |

## 8. 뒤로가기 (BackButton) 패턴

| 페이지 종류 | BackButton 필요? | 위치 / href |
|---|---|---|
| 워크스페이스 최상위 (`/chat`, `/documents`, ...) | 불필요 — 좌측 사이드바가 네비 담당 | — |
| 워크스페이스 깊은 페이지 (`/chatbots/[id]`, `/admin/metrics`) | 필요 | 헤더 좌측, 부모 list 페이지 href |
| 로그인 전 페이지 (`/login`, `/register`, `/manual`) | 필요 | 좌상단 fixed, BrandThemeGallery 와 대칭 |
| 모달 / SidePanel | 불필요 — ESC 또는 X 버튼 사용 | — |

링크 우선순위:
1. `href` 명시 (deep → list 같이 의도가 분명한 경우)
2. 생략하면 `router.back()` (브라우저 history)

라벨:
- 기본 "뒤로"
- 명시적 목적지 있으면 그 이름 (예: "로그인", "첫 화면", "로그인으로")
