# IA 가드 — 메뉴가 다시 늘어나지 않게 하는 룰

> 기능이 많아질수록 *발견성*은 떨어진다. 본 문서는 "메뉴 추가 = 인지 비용 발생"을
> 전제로, 신규 표면(Surface) 노출 전에 거치는 5단계 체크리스트를 정의한다.

## 핵심 원칙

1. **노출은 비용이다.** 메뉴 한 칸은 *모든* 사용자에게 0.5초 이상의 의사결정 부담을 부과한다 (Hick-Hyman). 추가하는 사람은 그 비용을 *제거할 가치*로 정당화해야 한다.
2. **라벨은 동사·결과형으로.** "AI 설정" ❌ → "상담봇 만들기" ⭕. "워크플로우" ❌ → "반복 작업 자동 실행" ⭕. 명사는 *개발 조직의 모듈명*이지 사용자의 *할 일*이 아니다.
3. **JTBD 단위로 묶는다.** 같은 사용자 의도(예: "지금 일을 끝낸다")에 속하는 기능은 같은 그룹에 있어야 한다. 개발 조직 구조를 IA로 옮기지 않는다.
4. **권한·고급 기능은 Surface 비노출이 기본.** 도구·스튜디오·관리·감사 등은 컨텍스트 진입(챗봇 빌더 안)·아바타 메뉴·⌘K 로만 도달.
5. **N ≤ 2 / 4.** 1차 결정(상단 그룹) 최대 2개, 2차(그룹 내 항목) 최대 4개.

## 현재 IA (2026-05 기준)

```
NavMenu (Surface)
├─ 일
│  ├─ 대화 시작           /chat
│  ├─ 챗봇 만들기         /chatbots
│  ├─ 자료 올리기         /documents
│  └─ 마켓에서 가져오기   /skills      (개발 전용 · 운영 미노출)
└─ 자동화                              (개발 전용 · 운영 미노출)
   ├─ 반복 작업 자동 실행 /workflows
   └─ 정해진 시간에 실행  /schedules

AvatarMenu (개인·도움·관리)
├─ 마이페이지 · 가이드
├─ 도움 & 커뮤니티 → 공지 · Q&A · FAQ
└─ (권한) 감사 · 관리

⌘K (전체 검색)
└─ NavMenu/AvatarMenu 전체 + 고급(도구·스튜디오, 개발 전용)
```

> **운영/개발 티어 (ADR-0009).** 운영(`NEXT_PUBLIC_FEATURE_SET=rag`) 빌드에서는 RAG 챗봇 4기능(대화·챗봇 만들기·자료 올리기·문서 검색)만 NavMenu 에 노출된다. `/skills`·`/workflows`·`/schedules`·`/tools`·`/studio`·`/workspaces` 는 개발 전용으로, 프론트는 `frontend/shared/lib/features.ts` 의 `DEV_ONLY_PATH_PREFIXES` + `isRagOnly` 가드로 메뉴·팔레트에서 숨기고, 운영 백엔드는 `backend/app/core/config.py` 의 feature 플래그(`skills_enabled`/`workflows_enabled`/`schedules_enabled`/`custom_tools_enabled`)에 따라 해당 라우터를 등록하지 않는다(404).

## 신규 메뉴 추가 전 5단계 체크리스트

추가하려는 메뉴 항목을 PR 설명에 **YES/NO**로 채워 첨부한다.

### 1. JTBD 검증
- [ ] **사용자의 한 줄 의도**로 라벨이 적혔는가? (동사·결과형)
- [ ] 기존 그룹("일" 또는 "자동화") 어디에도 못 들어가는가? — 들어갈 수 있으면 **신규 그룹 만들지 말 것**.
- [ ] 이미 있는 기능과 *결과*가 겹치지 않는가? (예: "스케줄" vs "워크플로"의 cron 트리거)

### 2. 발견성 vs 노출 비용
- [ ] 신규 사용자 100명 중 **30명 이상**이 첫 주에 이 메뉴를 클릭할 것으로 *근거 있게* 추정되는가?
  - 아니라면 → ⌘K / 컨텍스트 진입 / AvatarMenu 로.
- [ ] 메뉴를 추가했을 때 *다른 항목*의 클릭률이 떨어지지 않는가? (Cannibalization)

### 3. Progressive Disclosure
- [ ] 초보자에게도 *지금* 보여줘야 하는가? 아니면 N번 사용 후 노출이 적절한가?
- [ ] 관리자/팀장 전용이면 NavMenu가 아니라 **AvatarMenu의 관리 섹션**으로.
- [ ] "고급 기능"이면 **⌘K에만** (현재 정책: 도구·스튜디오는 ⌘K-only).

### 4. 라벨 동사화 테스트
- [ ] 라벨에 **추상 명사**가 있는가? — "설정·도구·자동화·AI·워크플로우·플랫폼·시스템" 들어가면 재작성.
- [ ] 라벨이 *결과/약속*을 말하는가? — "외부 앱 연결" 보다 "Slack에 알림 보내기"가 낫다.
- [ ] hint(보조 설명)가 *동사형 미니문장*인가?

### 5. 삭제 가능성
- [ ] 이 메뉴를 추가하면서 *어떤 기존 메뉴를 흡수·제거*할 수 있는가?
- [ ] 6개월 후에도 클릭률 < 1%면 **무조건 제거**한다고 약속할 수 있는가? (KPI 명시)

## 라벨 안티패턴 (절대 쓰지 말 것)

| 안티패턴 | 왜 나쁜가 | 대체 |
|---|---|---|
| 설정 | "내 계정"인지 "앱 환경설정"인지 모호 | 마이페이지 · 비밀번호 변경 |
| 도구 | 사용자에게 "도구"는 모든 것이다 | 외부 앱 연결 (그것마저 Surface 제거) |
| 자동화 / AI / 워크플로우 / 시스템 | 추상명사 — 결과를 말하지 않음 | 반복 작업 자동 실행 · 정해진 시간에 실행 |
| 마켓플레이스 | 영어 직역, 거리감 | 마켓에서 가져오기 |
| 대시보드 | 무엇의 대시보드인지 불명 | 이번 달 내 사용 · 팀 활동 기록 |
| 콘솔 | 개발자 용어 | 관리 · 팀 관리 |

| 감사 / 관리 | ❌ | ✅ 권한자에게만 | ✅ 권한자에게만 | – |
